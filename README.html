System architecture

3D Suite is structured as a web dashboard plus an API/backend layer. Browser-assisted workflows use Chromium/Playwright, while direct API workflows such as Gmail use OAuth tokens after initial authorization.

Run these commands from the 3D-SUITEG project root unless noted otherwise.

WeTransfer sender

The WeTransfer module uses browser automation and temporary mailbox providers to create/use a WeTransfer session and send files.

Current intended batch structure

One WeTransfer signup/browser session handles individual recipients sequentially, with session/account rotation after the configured batch boundary.

Temporary mailbox providers

The provider selection is passed from the dashboard to the backend. Existing integration work covered MailSlurp and Temp-Mail.io, with provider-aware mailbox creation and OTP polling.

Adobe Acrobat sender

Adobe is implemented as browser automation against the normal Acrobat web UI rather than relying on an Acrobat Sign API license.

Login

Adobe login happens in a normal browser session. The user signs in manually and the existing session is reused.

Sharing

Upload a PDF, wait for the document view, click Share, enter recipient(s), commit suggestions, invite, then return to the Acrobat files page for the next upload.

Important UI behavior already handled

  • Wait for Acrobat upload/document completion rather than relying only on a timeout.
  • Open the Share dialog after upload.
  • Handle the placeholder invite field that changes into the real invite input.
  • Use multiple selectors for Adobe invite fields.
  • Commit autocomplete suggestions for recipient addresses.
  • Return to https://acrobat.adobe.com/documents/files/ after each invite/send cycle.
Adobe account send limits Adobe account/plan limits are enforced by Adobe. Browser automation cannot increase the plan's allowed sends.

Gmail sender

Gmail is a separate sender module. Initial connection uses Google OAuth; sending uses the Gmail API with the connected account's saved authorization.

Existing Chromium profile

Scan manually, select a profile, then open Google OAuth in that profile.

Chromium profile + extension

Launch the selected profile and load an unpacked extension using Chromium's supported --load-extension behavior.

Manual OAuth URL

Generate an OAuth URL, open it in any browser/profile or even another PC, then paste the resulting callback URL back into 3D Suite.

Chromium profile handling

Profile scanning is manual. Friendly profile names are read from Chromium's profile metadata and mapped to actual folders such as Default, Profile 1, Profile 2, etc.

Typical Windows Chromium user data path
C:\Users\<username>\AppData\Local\Chromium\User Data

Friendly display names

A profile may appear as an email, person's name, or business name even though the disk folder is still Profile 3.

Already-open profiles

Chromium may reuse an already-running profile/process. Extension loading is most reliable when the extension is already installed or when 3D Suite launches the profile itself.

Gmail attachment modes

Attachment support is controlled by a master toggle. Only one attachment mode can be selected at a time.

Upload

Attach an existing file directly.

HTML → PDF

Render completed HTML through Chromium and create a single A4-style portrait PDF.

HTML → PPTX

Render HTML visually and place it into one A4-style portrait PowerPoint slide.

HTML → DOCX

Render HTML visually and fit it into one A4-style Word page.

HTML → SVG

Create one A4-style SVG canvas containing the completed HTML.

Single-page behavior

Current target
PDF   → 1 A4 portrait page
PPTX  → 1 A4-style portrait slide
DOCX  → 1 A4-style Word page
SVG   → 1 A4-style canvas
Visual fidelity vs native editability PPTX and DOCX preserve appearance by using a rendered image of the completed HTML. This keeps logos, colors, spacing, QR codes, and CSS looking like the browser render, but the underlying text is not recreated as fully native editable Word/PowerPoint elements.

Important Gmail status behavior

Gmail account status polling was changed to avoid terminal/dashboard flooding. Status is checked on panel load, after connect/disconnect, and when the user clicks Refresh accounts.

Troubleshooting

Gmail status returns 404

Verify apps/api/src/app/api/gmail/oauth/status/route.ts exists, then restart the development server.

OAuth URL still shows an old scope

Confirm the current connect route is installed and restart npm run dev. Generate a brand-new OAuth URL after restarting.

Request had insufficient authentication scopes

The connected token was authorized with fewer scopes than the route is attempting to use. Disconnect and reconnect after correcting the requested scope.

Manual OAuth callback fails on another PC

Expected when the redirect is localhost. Copy the full failed callback URL and paste it into the 3D Suite manual callback completion field.

Gmail account connects but does not appear in Send from

Check the Gmail status route, then click Refresh accounts. The account is stored by the API and read back through the status endpoint.

WeTransfer says MailSlurp while another provider is selected

Verify the selected provider is passed from credentials.wetransfer.provider and that the backend engine uses the unified provider implementation.

Adobe invite field times out

Adobe changes the invite input after the initial placeholder field is clicked. Use multiple selectors and wait for the real invite field before typing.

PPTX/DOCX looks too small

Current generated documents are configured as one A4-style portrait page/slide and proportionally fit the completed HTML. Keep the source HTML compact enough to remain readable when fitted to one page.

Security & operational notes

OAuth secrets

Google Client Secret and refresh tokens are sensitive. Do not post them publicly or commit them into Git.

Publishable keys

Logo.dev publishable keys are intended for client-visible usage, but still manage them deliberately and follow your provider plan terms.

Browser sessions

Existing Chromium/Adobe/Google sessions may contain sensitive cookies. 3D Suite should reuse browser sessions rather than exporting session cookies.

Provider limits

Gmail, Adobe, WeTransfer, Logo.dev, temporary mailbox providers, and Cloudflare all enforce their own account/API limits. Dashboard caps do not override provider limits.
